# video-sub-md 开发日志

---

## 1. 项目起源

### 原始需求

用户日常工作流中需要大量观看 Bilibili 和 YouTube 视频，但面临三个核心痛点：
1. 视频字幕无法批量保存，手动复制粘贴效率极低
2. 看完视频后难以快速提取核心内容做笔记
3. 英文字幕视频需要逐句翻译，没有趁手工具

### 核心动机

- **效率**：一次粘贴多个链接，批量下载，比逐个点开网页复制快 10 倍以上
- **知识管理**：字幕以 Markdown 输出，天然适合 Obsidian 等知识库工具
- **认知深化**：AI 四层分析协议（拓扑/语义/认知/批判）帮助用户不只是"看完"，而是"理解"
- **语言壁垒**：英中双语字幕翻译，降低非英语母语用户的学习门槛

## 3. 踩坑记录

### 坑 1：中文文件名被截断

| 项目 | 内容 |
|------|------|
| **问题** | 文件名如 `【中配】月入7.7万美元的独立开发者...` 被截断成 `【中配】月入7.md` |
| **根因** | `Path.with_suffix(".md")` 把中文中的 `.md` 误识别为原后缀，导致从 `.md` 处截断 |
| **解决方案** | 改用字符串拼接 `Path(str(output_path) + suffix)` 替代 `with_suffix()` |
| **涉及版本** | v0.2.0 |

### 坑 2：DeepSeek API 输出截断

| 项目 | 内容 |
|------|------|
| **问题** | AI 分析内容被截断，只输出 60-70% 就结束 |
| **根因** | `max_tokens=800` 不足以承载四层分析协议（约需 3000+ 中文字符） |
| **解决方案** | 逐步提升 max_tokens：800 → 4096 → 32768 |
| **涉及版本** | v0.2.0 → v0.3.0 |

### 坑 3：API 请求超时

| 项目 | 内容 |
|------|------|
| **问题** | DeepSeek API 调用时出现 timeout 错误 |
| **根因** | `timeout=60` 秒对于长输出不够，尤其在启用 reasoning 模式后 |
| **解决方案** | timeout 从 60 秒增加到 180 秒 |
| **涉及版本** | v0.3.0 |

### 坑 4：文件名字符非法

| 项目 | 内容 |
|------|------|
| **问题** | Windows 文件名保留字符 `\/:*?"<>|` 导致文件无法创建 |
| **根因** | 原方案将非法字符替换为下划线，但下划线仍可能造成语义不清 |
| **解决方案** | 直接删除非法字符，而非替换 |
| **涉及版本** | v0.2.0 |

### 坑 5：DeepSeek 模型弃用

| 项目 | 内容 |
|------|------|
| **问题** | deepseek-chat 和 deepseek-reasoner 将于 2026-07-24 弃用 |
| **根因** | DeepSeek API 官方公告，旧模型名称将逐步淘汰 |
| **解决方案** | 切换至 deepseek-v4-pro / deepseek-v4-flash，并添加 thinking 参数 |
| **涉及版本** | v0.3.0 |

### 坑 6：翻译行数错位（严重）

| 项目 | 内容 |
|------|------|
| **问题** | 62 行原文 → AI 返回 142 行译文 → 合并后译文全部错位，双语字幕对不上 |
| **根因** | AI（deepseek-v4-flash）未遵守"一行对一行"约束，长译文自动换行拆分成多行；原代码没有验证和修复机制 |
| **解决方案** | 三层修复：① Prompt 加全局编号 `[1]`~`[N]` 物理锁定行数 ② 解析时验证编号连续性 ③ `_force_line_alignment()` 强制合并/补齐 |
| **涉及版本** | v0.4.1 |

### 坑 7：翻译后丢失 frontmatter（严重）

| 项目 | 内容 |
|------|------|
| **问题** | 翻译后的 Markdown 文件丢失了标题、频道、链接、时长等元数据 |
| **根因** | `add_summary_to_file()` 和 `add_translation_to_file()` 都用 `split("---", 2)`，假设文件有 YAML frontmatter 成对边界（`---...---`），但实际文件中 `---` 只是一条 Markdown 分隔线。错误重建后 frontmatter 被损坏 |
| **解决方案** | 改为 `split("---", 1)`（只分割一次），第一部分为标题+元数据，第二部分为内容，分别处理后再拼接 |
| **涉及版本** | v0.5.0 |

### 坑 8：分析后翻译错位（严重）

| 项目 | 内容 |
|------|------|
| **问题** | 先分析后翻译时，分析内容的每一行都被当成了"原文"，消耗了本应分配给字幕的译文行，导致双语字幕错位 |
| **根因** | `merge_translation_to_bilingual()` 遍历文件 body 的每一行，非空行就取一个译文。当 body 中已有 `>` 开头的分析内容时，分析行也会消耗译文 |
| **解决方案** | `add_translation_to_file()` 新增 `original_subtitle` 参数，传入从原始文件中提前提取的纯字幕内容。双语合并时用这个纯字幕做 source，不受文件中已有分析内容干扰 |
| **涉及版本** | v0.5.0 |

### 坑 9：YouTube 标题含 `#` 导致 Obsidian 链接失效

| 项目 | 内容 |
|------|------|
| **问题** | YouTube 视频标题含 `#`（如 `#devsetup`）时，Obsidian URI 链接点击后提示"未找到文件" |
| **根因** | Obsidian URI 的 `file=` 参数把 `#` 解析为标题锚点分隔符（如 `file.md#header`），导致文件名被截断 |
| **解决方案** | `_safe_filename()` 中将 `#` 替换为 `_`。同时放弃 `Path.relative_to()`，改用大小写不敏感字符串前缀匹配，解决 Windows 盘符/大小写差异 |
| **涉及版本** | v0.5.0 |
| **误判成本** | 最初怀疑是 VS Code 更新导致 OSC 8 超链接失效（浪费 30 分钟），后怀疑是 `Path.relative_to()` 问题（浪费 15 分钟），最终通过"Bilibili 能点、YouTube 不能点"的对比发现是数据差异（`#` 字符） |

### 坑 11：BILI_COOKIE 环境变量覆盖 config.py（严重）

| 项目 | 内容 |
|------|------|
| **问题** | 用户已更新 `config.py` 中的 `DEFAULT_SESSDATA`，但工具仍使用旧 Cookie，提示“未通过 B站登录校验”，B站字幕全部下载失败 |
| **根因** | 代码读取优先级为：`--cookie` > `BILI_COOKIE` 环境变量 > `BILIBILI_SESSDATA` 环境变量 > `config.py`。用户使用的第三方启动器菜单在启动终端时带入了缓存的旧 `BILI_COOKIE`，导致 `config.py` 被覆盖 |
| **解决方案** | ① 删除系统/用户环境变量中的 `BILI_COOKIE` ② 在启动器 BAT 中加入 `set "BILI_COOKIE="` 清空当前进程的环境变量 ③ 提供 `run.ps1` / `run.cmd` 作为兜底启动方式，自动清空 `BILI_COOKIE` |
| **涉及版本** | v0.5.4 |
| **5Why 根因** | 为什么 config.py 不生效？→ 因为环境变量优先级更高。为什么启动器会带旧环境变量？→ 因为启动器进程或其父进程在启动时继承了旧的 `BILI_COOKIE`。为什么旧值还在？→ 因为用户虽然改了系统环境变量，但已运行的启动器进程不会自动刷新环境变量。系统级根因：环境变量具有进程继承性，运行中的进程不会监听注册表变化 |

### 坑 10：config.py 敏感信息误提交

| 项目 | 内容 |
|------|------|
| **问题** | `config.py` 包含真实 API Key 和 SESSDATA，有被意外提交到 GitHub 的风险 |
| **根因** | 项目初期没有配置分离机制，敏感信息和业务代码混在同一文件 |
| **解决方案** | ① 创建 `config.example.py` 作为模板（含空占位符和注释）② 将真实 `config.py` 加入 `.gitignore` ③ 在 README 中明确标注配置方式 |
| **涉及版本** | v0.5.0 |
| **5Why 根因** | 为什么 config.py 会被提交？→ 因为没有 `.gitignore` 规则。为什么没有规则？→ 因为项目初期只考虑了功能实现，没有考虑安全合规。为什么没有考虑？→ 因为缺乏配置分离层的架构设计。系统级根因：项目缺乏"配置层抽象"，将环境配置与业务代码耦合 |

## 4. 设计决策

### 决策 1：循环下载模式 vs 单次执行

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~每次运行只处理一批~~ | ❌ 放弃 | 用户需要持续下载多个批次，频繁重启程序体验差 |
| 采用 while True 循环 | ✅ 采用 | 按 q 退出，保持程序常驻，适合批量工作流 |

### 决策 2：AI 分析协议设计

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~一次性输出所有分析~~ | ❌ 放弃 | 用户场景多样（教学/观点/混合），单一协议难以覆盖 |
| 采用四层协议 + 条件激活插件 | ✅ 采用 | 通用层（必执行）+ 类型层（条件激活），兼顾全面性和针对性 |
| 元认知安全锁置顶 | ✅ 采用 | 防幻觉是 AI 分析的生命线，必须最高优先级 |
| 不强制数量要求 | ✅ 采用 | 避免 AI 为凑数而拆分或编造内容 |
| 趣味叙事重构 | ✅ 采用 | 降低阅读门槛，让分析不只是学术堆砌 |

### 决策 3：文件名处理策略

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~替换为下划线~~ | ❌ 放弃 | 下划线可能与原有语义混淆，如 `a/b` → `a_b` 改变了原意 |
| 直接删除非法字符 | ✅ 采用 | 语义损失最小，且 Windows 本身不支持这些字符 |
| 保持原始长度 | ✅ 采用 | FILENAME_MAX_LENGTH 从 80 扩展到 150，适应长标题视频 |

### 决策 4：翻译模型选择

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~用 Pro 模型翻译~~ | ❌ 放弃 | 翻译是简单重复任务，Pro 成本高、速度慢，大材小用 |
| 使用 Flash 模型翻译 | ✅ 采用 | Flash 速度更快、成本更低，足够完成字幕翻译 |
| ~~翻译后合并到分析~~ | ❌ 放弃 | 用户场景多样，分析和翻译需求独立，不应强制绑定 |
| 双询问机制 | ✅ 采用 | 用户可自由组合：分析/翻译/两者/都不，灵活度最高 |

### 决策 5：翻译对齐策略

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~信任 AI 自觉遵守行数~~ | ❌ 放弃 | deepseek-v4-flash 轻量模型不听话，62 行原文返回 142 行 |
| ~~放弃编号，靠顺序匹配~~ | ❌ 放弃 | AI 返回空行或废话时顺序会错位 |
| 全局连续编号 + 强制对齐 | ✅ 采用 | 编号物理锁定行边界，偏差时自动合并修复，鲁棒性最高 |

### 决策 6：Frontmatter 处理策略

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~用 `split("---", 2)` 解析~~ | ❌ 放弃 | 假设 `---` 是 YAML frontmatter 成对边界，但实际文件中 `---` 只是一条 Markdown 分隔线，导致 frontmatter 被错误重建 |
| `split("---", 1)` + 分别处理 | ✅ 采用 | 只分割一次，第一部分=标题+元数据，第二部分=内容，各自处理后拼接，不破坏任何现有格式 |

### 决策 7：分析与翻译共存策略

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~直接读取文件当前内容做翻译~~ | ❌ 放弃 | 文件可能已有分析内容，会导致分析行消耗译文行 |
| 传入 `original_subtitle` 参数 | ✅ 采用 | 在添加分析之前保存原始字幕内容，翻译时用这个纯内容做 source，不受后续分析内容干扰 |
| 在文件中插入 HTML 标记分隔 | ❌ 放弃 | 引入 `<!-- SUBTITLE_START -->` 等标记会增加文件复杂度，用户可能不喜欢 |

### 决策 8：启动器环境变量覆盖问题

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~降低环境变量优先级，让 config.py 优先~~ | ❌ 放弃 | 会破坏既有用户习惯和 README 文档约定，且环境变量在纯命令行场景下确实更安全 |
| ~~移除环境变量支持~~ | ❌ 放弃 | 会损失灵活性，部分用户依赖 CI/脚本注入 Cookie |
| 保持现有优先级 + 提供启动脚本清空 `BILI_COOKIE` | ✅ 采用 | 不破坏兼容性；为使用第三方启动器的用户提供 `run.ps1` / `run.cmd` 兜底方案；在启动器 BAT 中显式 `set "BILI_COOKIE="` 是局部、低侵入的修复 |
| 增加启动时 Cookie 有效性检测 | ✅ 采用 | 让用户第一时间知道当前生效的 Cookie 是否有效，减少“为什么有字幕却下载失败”的困惑 |

### 决策 9：配置安全策略

| 选项 | 决策 | 理由 |
|------|------|------|
| ~~直接在代码中写死 API Key~~ | ❌ 放弃 | 极不安全，一旦提交 GitHub 就无法撤回 |
| ~~只依赖 `.gitignore`~~ | ❌ 放弃 | `.gitignore` 可能被人为修改或忽略，不是可靠的安全机制 |
| `config.example.py` + `.gitignore` + 文档提醒 | ✅ 采用 | 三层防护：模板文件中没有真实值、真实配置文件被 Git 忽略、README 明确标注配置方式。即使两层失效，第三层仍能兜底 |
| ~~引入 pydantic-settings~~ | ⏳ 未来 | v0.6.0 计划引入，支持 `.env` 文件和环境变量自动加载，进一步降低误配置风险 |
